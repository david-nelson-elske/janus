## EntityStore interface

The primary store interface, updated to unify `browse()` and `get()` into `read()`:

```ts
interface EntityStore {
  // Unified read
  read(entity: string, params?: ReadParams): Promise<EntityRecord | ReadPage>;

  // Single-record writes
  create(entity: string, record: NewEntityRecord): Promise<EntityRecord>;
  update(entity: string, id: string, patch: Partial<EntityRecord>): Promise<EntityRecord>;
  delete(entity: string, id: string): Promise<void>;

  // Bulk writes
  createMany(entity: string, records: readonly NewEntityRecord[]): Promise<EntityRecord[]>;
  updateWhere(entity: string, where: WhereClause, patch: Partial<EntityRecord>): Promise<number>;
  deleteWhere(entity: string, where: WhereClause): Promise<number>;

  // Utilities
  exists(entity: string, id: string): Promise<boolean>;
  count(entity: string, where?: WhereClause): Promise<number>;

  // Transactions
  withTransaction<T>(fn: (tx: EntityStore) => Promise<T>): Promise<T>;
}
```

### read() parameterization

```ts
interface ReadParams {
  readonly id?: string;
  readonly where?: WhereClause;
  readonly sort?: readonly SortClause[];
  readonly window?: ReadWindow;
  readonly include?: readonly IncludeSpec[];
  readonly search?: string;
}

interface ReadWindow {
  readonly limit: number;
  readonly offset?: number;
  readonly cursor?: string;
}

interface ReadPage {
  readonly records: readonly EntityRecord[];
  readonly total: number;
  readonly window: ReadWindow;
  readonly hasMore: boolean;
}
```

**Behavior:**
- `read('note', { id: '123' })` → returns `EntityRecord | null`. Single-record lookup by id.
- `read('note')` → returns `ReadPage`. All records with default pagination.
- `read('note', { where: { status: 'published' } })` → returns `ReadPage`. Filtered list.
- `read('note', { id: '123', include: [...] })` → Single record with relation includes.

The return type is `EntityRecord | ReadPage`. When `id` is provided, the return is a single record (or null). When `id` is absent, the return is a `ReadPage`. The CRUD handler ([see below](#crud-handlers)) normalizes this into the `ConcernContext.result` shape.

**Singleton handling:** For Singleton() entities, `read('config')` (no params) returns the single record. `read('config', { id })` also works — the singleton id is derived from the entity name.

### Types carried forward

```ts
// Unchanged from packages/next
interface EntityRecord {
  readonly id: string;
  readonly [key: string]: unknown;
}

interface NewEntityRecord {
  readonly [key: string]: unknown;
  // id is optional — auto-generated if absent
  readonly id?: string;
}

interface WhereClause {
  readonly [field: string]: unknown | { $gt?: unknown; $lt?: unknown; $in?: unknown[]; /* ... */ };
}

interface SortClause {
  readonly field: string;
  readonly direction: 'asc' | 'desc';
}

interface IncludeSpec {
  readonly relation: string;
  readonly fields?: readonly string[];
}
```

## persist_routing

The `persist_routing` records ([generated by compile](03-compile-and-dispatch-index.md)) drive adapter selection:

```ts
interface RoutingRecord {
  readonly entity: string;
  readonly table: string;
  readonly adapter: 'relational' | 'memory' | 'file' | 'derived' | 'virtual';
  readonly schema: Readonly<Record<string, unknown>>;
  readonly storage: StorageStrategy;
}
```

`createEntityStore()` reads these records and routes each entity to the correct adapter:

```ts
function createEntityStore(config: {
  routing: readonly RoutingRecord[];
  adapters: {
    relational?: StoreAdapter;
    memory?: StoreAdapter;
    file?: StoreAdapter;
    derived?: StoreAdapter;
    virtual?: StoreAdapter;
  };
}): EntityStore
```

The returned `EntityStore` dispatches each call to the adapter specified by the entity's routing record. If an entity is not found in routing, the call throws `UnknownEntityError`.

## StoreAdapter interface

Individual adapter implementations satisfy this interface (carried forward from packages/next):

```ts
interface StoreAdapter {
  read(meta: AdapterMeta, params?: ReadParams): Promise<EntityRecord | ReadPage>;
  create(meta: AdapterMeta, record: NewEntityRecord): Promise<EntityRecord>;
  update(meta: AdapterMeta, id: string, patch: Partial<EntityRecord>): Promise<EntityRecord>;
  delete(meta: AdapterMeta, id: string): Promise<void>;

  createMany(meta: AdapterMeta, records: readonly NewEntityRecord[]): Promise<EntityRecord[]>;
  updateWhere(meta: AdapterMeta, where: WhereClause, patch: Partial<EntityRecord>): Promise<number>;
  deleteWhere(meta: AdapterMeta, where: WhereClause): Promise<number>;

  exists(meta: AdapterMeta, id: string): Promise<boolean>;
  count(meta: AdapterMeta, where?: WhereClause): Promise<number>;

  initialize(entities: readonly AdapterMeta[]): Promise<void>;
  shutdown(): Promise<void>;
}

interface TransactionalAdapter extends StoreAdapter {
  withTransaction<T>(fn: (adapter: StoreAdapter) => Promise<T>): Promise<T>;
}

interface AdapterMeta {
  readonly entity: string;
  readonly table: string;
  readonly schema: Readonly<Record<string, unknown>>;
  readonly storage: StorageStrategy;
}
```

## Adapter implementations

All adapters are carried forward from packages/next with the `browse()`/`get()` → `read()` API change:

| Adapter | Storage strategies | Source |
|---------|-------------------|--------|
| Memory adapter | Volatile, Singleton (cache) | `packages/next/store/memory-adapter.ts` |
| SQLite adapter | Persistent, Singleton | `packages/next/store/sqlite/` |
| File adapter | Persistent (file-backed) | `packages/next/store/file-adapter.ts` |
| Derived adapter | Derived | `packages/next/store/derived-adapter.ts` |
| Cache wrapper | Singleton overlay | `packages/next/store/cache-adapter.ts` |

The internal implementation changes are minimal — rename `browse()` → `read()` (when no id), `get()` → `read()` (when id present), unify the method.

## CRUD handlers

Four execution records registered as the core operation handlers. Each is an `ExecutionHandler` ([02](02-wiring-functions.md)) that operates on the `ConcernContext`:

### store-read

```ts
const storeRead: ExecutionHandler = async (ctx) => {
  const params = ctx.parsed as ReadParams | undefined;
  const result = await ctx.store.read(ctx.entity, params);

  if (params?.id) {
    // Single record
    if (!result) throw new NotFoundError(ctx.entity, params.id);
    ctx.result = { kind: 'record', record: result as EntityRecord };
  } else {
    // Page
    ctx.result = { kind: 'page', page: result as ReadPage };
  }
};
```

### store-create

```ts
const storeCreate: ExecutionHandler = async (ctx) => {
  const record = await ctx.store.create(ctx.entity, ctx.parsed as NewEntityRecord);
  ctx.result = { kind: 'record', record };
};
```

### store-update

```ts
const storeUpdate: ExecutionHandler = async (ctx) => {
  const { id, ...patch } = ctx.parsed as { id: string; [key: string]: unknown };
  ctx.before = await ctx.store.read(ctx.entity, { id }) as EntityRecord | null;
  const record = await ctx.store.update(ctx.entity, id, patch);
  ctx.result = { kind: 'record', record };
};
```

### store-delete

```ts
const storeDelete: ExecutionHandler = async (ctx) => {
  const { id } = ctx.parsed as { id: string };
  ctx.before = await ctx.store.read(ctx.entity, { id }) as EntityRecord | null;
  await ctx.store.delete(ctx.entity, id);
  ctx.result = { kind: 'void' };
};
```

### PersistResult

```ts
type PersistResult =
  | { readonly kind: 'record'; readonly record: EntityRecord }
  | { readonly kind: 'page'; readonly page: ReadPage }
  | { readonly kind: 'output'; readonly data: unknown }
  | { readonly kind: 'void' };
```

This is set on `ctx.result` by the core handler and consumed by the respond adapter ([05](05-pipeline-concern-adapters.md)).

### Singleton handling

For Singleton entities, the `update` handler resolves the singleton id from the entity name (convention: entity name IS the singleton id). The `read` handler returns the single record when no id is provided.

### Bulk operations

Bulk CRUD is exposed on the `EntityStore` interface for internal use (subscription handlers, migration tools) but is not wired as separate pipeline concerns. A bulk create, for example, would be a custom action that calls `ctx.store.createMany()` directly.

## Testing gate

When 124-04 is implemented, the following should be testable:

- `read('note', { id: '123' })` returns single `EntityRecord`
- `read('note')` returns `ReadPage` with all records
- `read('note', { where: { status: 'published' } })` returns filtered `ReadPage`
- `read('config')` on a Singleton returns the single record
- `create('note', { title: 'Test' })` creates and returns record with generated id
- `update('note', '123', { title: 'Updated' })` updates and returns record
- `delete('note', '123')` removes record
- `createMany('note', [...])` creates multiple records
- `updateWhere('note', { status: 'draft' }, { status: 'archived' })` bulk updates
- Store router dispatches to correct adapter per `persist_routing`
- Memory adapter passes all tests
- SQLite adapter passes all tests
- Derived adapter only supports `read()`
- `withTransaction()` wraps writes in a transaction (SQLite)
- CRUD handlers set `ctx.result` correctly
- CRUD handlers set `ctx.before` for update/delete operations
- Unknown entity throws `UnknownEntityError`
